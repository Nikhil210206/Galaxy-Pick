Galaxy-Pick

PJ1 · Samsung Galaxy Phone Recommendation System — Team Execution Plan

Module: Samsung GenAI Capstone (PJ1)  •  Team: Member 1–4  •  Build window: 3–5 days  •  Date: 14 Jul 2026

The decision, in one line. Build a deterministic Galaxy phone recommender — a curated 2024–2026 dataset → four transparent 0–10 scores → a Weighted Sum Model → an explained top-3 — delivered as a Jupyter notebook (the graded core) plus a Samsung-styled Streamlit app, with an optional, free-tier Gemini layer for natural-language input that always falls back to an offline rule-based parser. It runs with no API key and no network.

Executive summary (one page — read this first)

3 · Dataset strategy

Verdict on the source data: the S-/Z-series rows in the team CSV are real and good — keep them. The A/M/F rows (A12, A14…) are fabricated arithmetic padding with wrong labels — drop them. The Kaggle scrapes are 2022–23, so with a 2024–2026 catalog they no longer match — demote to a calibration reference only.

Key insight — more phones does not help. A WSM has no training step, so extra rows can't raise accuracy — they only add noise. Curate, don't accumulate.

Recommended architecture — one row per model:

  • Range: 2024–2026. 2024–25 real; every 2026 row is spec_source=mock (a projection), flagged honestly.
  • Size: ~36 models (27 real / 9 mock) — spans budget → flagship → foldable.
  • Images: free, offline placeholder cards drawn in code (zero copyright risk).
  • Columns: identity · raw specs · four scores · spec_source.

QuestionDecisionWhy

RAG / Vector DBNo36 rows fit in a DataFrame; nothing to retrieve.

Grok / HFNoPaid / off-theme / extra keys, zero added value.
The one standout AI feature. A free-text box ("a phone for photography under ₹50k") → Gemini returns JSON weights if a key exists, else a deterministic keyword parser does the same. Output is validated before use. Showcases Ch.2–4 and guarantees the demo can't break.

7 · UI / UX strategy

  • Layout: sidebar (persona dropdown + free-text + budget slider) → main: top-3 cards → ranked table → score-breakdown bar.
  • Samsung language: Samsung Blue #1428A0, One-UI rounded cards, generous whitespace, clean sans type.
  • Accessibility: strong contrast, alt text, never color-only meaning (always show the number), keyboard-navigable.
  • On Stitch: fine as a mockup aid, but don't adopt a full React stack — it abandons the required Jupyter environment. Streamlit gives better polish-per-hour.

12 · Creativity opportunities (on-theme, not gimmicks)

  • Live weight sliders — reorder results in real time; makes the WSM self-evident in the room.
  • Contrastive "why not X" — explains the phone it didn't pick (over budget / weaker on your top factor).
  • Close-call flag — when the top two are within a small margin, say so (honest UX = Ch.4).
  • Provenance & limitations panel — real vs. mock, scoring caveats; judges reward visible Responsible AI.

13 · Final recommendation

Build Version B: a deterministic Galaxy-Pick engine (curated 2024–2026 dataset → four documented 0–10 scores → WSM → explained top-3), as a Jupyter notebook (graded core) + a Samsung-styled Streamlit app, with an optional free-tier Gemini layer that always degrades to a deterministic parser.

Why over the alternatives: the big production stack scores nothing here, costs days you don't have, and signals you misread a prompt-engineering / Responsible-AI exercise. Version A merely passes; B adds the differentiators (natural-language input, explainability, a Responsible-AI story) within the 3–5 day window; C's flourishes drop into B only if time remains. Deterministic-core-first means the project is fully demonstrable with zero API dependency — Gemini becomes a clean bonus, never a single point of failure.
